## Step 0 sanity check — port-8000 conflict + pushback moment

`docker compose up` failed immediately: port 8000 was already bound by
portainer on Evan's host. README hard-codes `http://localhost:8000`.

I surfaced three options to Evan:
1. Local-only `docker-compose.yml` tweak to `8088:8000` (don't commit;
   reviewers fresh-cloning still get 8000) ← **my recommendation**
2. Stop portainer for the session
3. Permanently change to 8088 + update the README ← flagged not-recommended

Evan pushed back: **"No. Make port configurable via .env (and log this
as a decision where I pushed back)."**

He was right. My instinct was budget-paranoid (smallest local change).
His was reviewer-paranoid (anyone running portainer/jenkins/airflow hits
the same wall). `.env` is the idiomatic answer, default unchanged, ~10
lines of total change. Captured the full pushback rationale in
[[decision-port-configurable]] — `status: approved`.

Implementation:
- New `.env.example` (committed) documents `FOLIO_PORT` and the default
- New `.env` (gitignored) holds the per-developer override
- `docker-compose.yml` uses `${FOLIO_PORT:-8000}` for the host port and
  passes `FOLIO_PORT` into the container
- `seed.php` reads `getenv('FOLIO_PORT')` to print the right URLs
- `.gitignore` adds `.env`
- README documents the override in one short sentence

Default flow (`docker compose up` from a fresh clone) is unchanged.
My `.env` sets `FOLIO_PORT=8088`.

This is video material. First real pushback moment; demonstrates exactly
the "where AI helped or hurt, where you pushed back" beat the README
asks for explicitly.
