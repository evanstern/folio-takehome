## Step 0 sanity check — port-8000 conflict + pushback moment

`docker compose up` failed immediately: port 8000 was already bound by
portainer on Evan's host. README hard-codes `http://localhost:8000`.

I surfaced three options to Evan:
1. Local-only `docker-compose.yml` tweak to `8088:8000` (don't commit;
   reviewers fresh-cloning still get 8000) ← **my recommendation**
2. Stop portainer for the session
3. Permanently change to 8088 + update the README ← flagged not-recommended

Evan pushed back: **"No. Make port configurable via .env (and log this
as a decision where I pushed back)."**

He was right. My instinct was budget-paranoid (smallest local change).
His was reviewer-paranoid (anyone running portainer/jenkins/airflow hits
the same wall). `.env` is the idiomatic answer, default unchanged, ~10
lines of total change. Captured the full pushback rationale in
[[2026-05-18-1842-decision-port-configurable]] — `status: approved`.

Implementation:
- New `.env.example` (committed) documents `FOLIO_PORT` and the default
- New `.env` (gitignored) holds the per-developer override
- `docker-compose.yml` uses `${FOLIO_PORT:-8000}` for the host port and
  passes `FOLIO_PORT` into the container
- `seed.php` reads `getenv('FOLIO_PORT')` to print the right URLs
- `.gitignore` adds `.env`
- README documents the override in one short sentence

Default flow (`docker compose up` from a fresh clone) is unchanged.
My `.env` sets `FOLIO_PORT=8088`.

This is video material. First real pushback moment; demonstrates exactly
the "where AI helped or hurt, where you pushed back" beat the README
asks for explicitly.

## Bare-layout migration + docker compose project-name pin

Mid-session Evan directed: **"this workspace needs to be a .bare repo
as well."** Ran `coda-lite repo_bare_init` and converted the project to
the bare layout — `.bare/` + sibling worktrees, with `main/` as the
worktree civicplus operates from. Captured in [[2026-05-18-1848-decision-bare-layout]].

Immediate fallout: docker compose derives its project name from the
working directory, so the container went from `folio-takehome-app-1`
to `main-app-1`. Evan caught it instantly: **"docker compose project
name cannot change. or at least not on `main`."**

Fix: pinned `name: ${COMPOSE_PROJECT_NAME:-folio-takehome}` at the top
of `docker-compose.yml`. Default matches what a fresh-clone reviewer
expects; feature worktrees can override via `COMPOSE_PROJECT_NAME` in
their `.env` if they want parallel docker runs. Captured in
[[2026-05-18-1851-decision-compose-project-name-pinned]].

Side effects of the migration that I touched:
- `civicplus` symlink at `~/agents/civicplus` repointed from
  `.../folio-takehome/.coda` to `.../folio-takehome/main/.coda`
- `pattern-branching-worktrees` updated to describe the new
  `~/projects/folio-takehome/<slug>/` layout (was the old
  parallel-to-project-root sibling layout)
- `folio-docker` entity page updated to reflect the pinned `name:`
- SOUL.md "Repositories" block updated
- `db.sqlite` now lives at `main/db.sqlite` (gitignored, no impact)

Smoke test post-fix: `docker compose ps` shows `folio-takehome-app-1`,

## Wiki slug convention: date-prefix everything

Evan flagged in two consecutive turns: bare slugs make ordering
illegible, and patterns specifically matter because they're operational
backbone (his words: "important for civic plus hirering mgmt").

**New rule, codified in `wiki/index.md` "Slug naming" section:** every
new wiki page is `YYYY-MM-DD-HHMM-<slug>.md`. Minute precision, local
time. Exceptions documented inline:
- `incidents/flag-*.md` — persistent attributes, not events; no ordering
- Baseline entity pages seeded by Zach (`folio-app`, `folio-schema`,
  `folio-bootstrap`, `folio-admin-page`, `folio-share-page`,
  `folio-view-page`, `folio-tests`, `folio-docker`) — the codebase
  map, not session timeline

Retro-applied to:
- All four pages civicplus added today using real timestamps:
  `2026-05-18-1836-folio-layout`, `2026-05-18-1842-decision-port-configurable`,
  `2026-05-18-1848-decision-bare-layout`,
  `2026-05-18-1851-decision-compose-project-name-pinned`
- All five Zach-seeded decisions with synthetic pre-boot timestamps
  reflecting documented order: `1620-decision-coda-dir-shipped`
  (the meta-decision, approved from the start), then the four
  proposed decisions at 16:30/16:35/16:40/16:45 matching PROJECT.md's
  strategy order (migrations → scheduling → readable-ids → search)
- All six patterns: Zach's original three at 16:00/16:05/16:10
  (audit-log, test-harness, feature-session-brief — pre-Evan-review),
  the three Evan-driven additions at 17:30/17:35/17:40
  (branching-worktrees, focus-task-convention, collaboration-with-evan
  — post-edits, in the order learnings/2026-05-18-evan-edits-to-civicplus.md
  describes them being written)

Method: `git mv` for tracked files, plain `mv` for the two uncommitted
new decisions, then a single `sed -e ... -e ...` pass over every file
containing the old `[[bare-slug]]` references to rewrite them.

Verification: `grep -rn '\[\[\(pattern-\|decision-\|folio-layout\)\]\]'
.coda/` returns nothing. No leftover bare references anywhere.

The payoff: `ls .coda/wiki/decisions/` now reads as a timeline —
Zach's baseline cluster (16:20–16:45), gap, civicplus's session work
(18:42–18:51). Same for patterns: Zach's original three pre-Evan,
his three Evan-driven adds post-review. A reviewer (or future
civicplus rebooting cold) can see ordering and rough dependency
chains without opening files.

Lesson for civicplus: when you create a new wiki page, the timestamp
is `date +%Y-%m-%d-%H%M` at creation moment. Set it once in the
filename and once in frontmatter `created:`. Bump frontmatter
`updated:` on subsequent edits but NEVER rename the file once it's
landed — the wikilinks are everywhere.
